# 全盘审查与系统级强化 · 2026-05-23 (v2)

## 审查方法

并行扫描代码库（grep + readCode + 文件 stats），聚焦四个维度：
- **算法/统计严谨性** — calibration / leak / 共形保证
- **代码质量** — 单文件行数 / 重复硬编码 / try-catch 静默
- **UX 完整度** — 双端对称性 / 解释性 / 长期价值
- **工程纪律** — SW 维护 / 依赖 / 错误兜底

## 发现

### A. 算法层（严重）

**A1. Calibration T 触界**
demo 模型里 SSQ blueT = `9.999952` / DLT blueT = `9.999952`（正好顶到上界 10），
说明黄金分割搜索在这个方向 NLL 是平的。这是"模型对该 head 几乎无信号"的信号
被误读为"找到了最优 T"。

**A2. In-sample calibration leak**
T 在 val set fit，回测又在同一 val set 报 ECE before→after。这个改善是 in-sample
拟合，不是泛化。

**A3. 缺乏 conformal prediction**
温度校准只修概率"锐度"。没有任何工具能给"hit@K 集合频率主义保证"。

**A4. Switch 静默覆盖 default key**
用户从 manager 切换到模型 X，会把 X 的 payload 覆写到 default key，原 default 数据丢失。

### B. 代码质量（中）

**B1. Controller 单文件过大**
`lstm-controller.js` 1064 行 / `dlt-lstm-controller.js` 853 行，违反 SRP。

**B2. SW APP_SHELL 手动维护**
新加 JS 文件忘了登记会导致离线缺资源。

**B3. 重复硬编码 6×6/33 1/16**
散在 6+ 个文件，没有 BASELINES 单一来源。

**B4. SSQ 缺号码体检**
DLT 有完整 dlt-explainer.js 6 维度评分，SSQ 没有，双端不对称。

### C. UX（中）

**C1**. A/B 对比工作台缺 reliability 双线 + 共形覆盖率
**C2**. 没有"我做的预测真实命中率追踪"——长期价值最强的功能
**C3**. SeqLen 敏感性 / 训练曲线对比次于上述

### D. 工程（轻）

**D1**. tools/check_*.mjs 未进 CI
**D2**. puppeteer-core 用 floating 版本
